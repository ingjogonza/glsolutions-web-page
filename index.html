02

El problema

¿Tu código crece,
pero tu equipo lo entiende
cada vez menos?

→

La IA genera código rápido, pero nadie sabe qué rompe qué.

→

Si el dev que construyó el sistema se va, el conocimiento se va con él.

→

Pagaste por desarrollo y no puedes auditar ni explicar lo que recibiste.

→

Los releases se hacen más lentos aunque el equipo trabaja igual.

→

Un inversor técnico pide ver la arquitectura y nadie puede explicarla.

Estos problemas tienen un origen común: el sistema se construyó sin una especificación que lo defina primero. La IA genera rápido, pero sin spec genera sin contexto de negocio.

Existe una forma de construir software donde cada decisión técnica queda documentada antes de escribir una línea de código.

03

La solución

Spec Driven Development —
la spec define el sistema
antes del código.

SDD es la metodología con la que GL Solutions construye software. Cada decisión técnica queda documentada antes de escribir una línea de código. La IA ejecuta lo que la spec ordena — no al revés. El resultado: un sistema que puedes auditar, transferir y escalar sin depender de una sola persona.

Vibe Coding

  • IA genera código sin contexto de negocio
  • Sin especificaciones → sin trazabilidad
  • Deuda técnica invisible hasta que escala
  • Imposible hacer handoff a otro equipo
vs

Spec Driven Development

  • La spec define el sistema antes del código
  • La IA implementa lo que la spec ordena
  • Cada decisión técnica queda documentada
  • Cualquier equipo puede continuar el proyecto
01

Spec antes que código

Ninguna línea de código sin una especificación que la justifique. La spec es la fuente de verdad del sistema.

02

La IA obedece, no decide

Las decisiones de arquitectura, dominio y trade-offs técnicos las toman los ingenieros. Siempre.

03

Todo artefacto es auditable

Cualquier ingeniero externo puede entender el sistema leyendo sus artefactos. El handoff limpio no es opcional.

05

Cómo funciona

Un proceso en 5 fases.
Cada una tiene una
puerta de calidad.

F1

Discovery & Domain Modeling

Entendemos el negocio antes de tocar el teclado. Glosario de dominio, event map y mapa de actores. El cliente aprueba antes de avanzar.

3–5 días
F2

Architecture & API Design

Decidimos la estructura antes de implementarla. ADRs, diagrama C4 y OpenAPI spec. Cada decisión técnica documentada con alternativas descartadas.

4–6 días
F3

Feature Specification

Definimos qué hace el sistema antes de construirlo. Criterios BDD, edge cases registrados y contrato de API por feature. El cliente firma el alcance.

3–5 días
F4

Spec-Guided Implementation

La IA construye lo que la spec ordena. Code review contra criterios BDD. 100% de criterios de aceptación cubiertos por tests automatizados.

Iterativo
F5

Validation & Handoff

Entregamos un sistema que cualquier ingeniero puede operar desde el día uno. UAT firmado, runbook, guía de onboarding y arquitectura viva.

3–4 días
El rol de la IA

Usamos IA donde acelera sin perder control. La IA genera borradores de specs, escribe código desde la feature spec y produce tests desde BDD. Las decisiones de arquitectura, la priorización del backlog y la aprobación de que el sistema cumple el negocio las tomamos siempre los ingenieros.
